WEAKNESS-1 [MAJOR]: The 30%-drop success threshold is plucked from thin air and the candidate knows it. — The PR is conditioned on "median `cost_tokens` for shell-mode runs drops ≥30%" but there is no derivation of why 30 and not 10 or 50, no priors from the existing `phase1-report.json` showing what fraction of token cost comes from the branch-3 escalation path, and no analysis of how many of the existing scenarios even *exercise* non-zero exits from informational commands. If the existing bench has only one or two scenarios where `INFORMATIONAL_NONZERO` could possibly fire, "median drops 30%" is either trivially achievable (by inflating those rows with the four new cherry-picked scenarios) or impossible regardless of how good the change is. The "no human judgement" framing is the tell — a hand-tuned threshold pretending to be falsifiable.

WEAKNESS-2 [MAJOR]: The benchmark is being used as both the training set and the test set. — Step 4 says "add 4 scenarios — `cmd_aware_git_diff_clean`, `cmd_aware_grep_miss`, `cmd_aware_test_false`, plus one negative" and step 1 says the `INFORMATIONAL_NONZERO` set is `{"git diff --quiet", ..., "grep", ..., "test", ...}`. The author is literally writing the eval scenarios for the same commands they whitelisted. The success metric ("over the union of existing phase1 scenarios plus the four new ones") then folds those new scenarios into the median. This is target leakage dressed up as a falsifiable experiment, and it directly contradicts the candidate's own claim of "no opt-in corpus, no redaction policy required."

WEAKNESS-3 [FATAL]: The "needs_llm=False" classification for non-zero exit on whitelisted commands silently swallows real failures. — `git diff --quiet` is whitelisted but `git diff --quiet` exits with **2** on usage error or pathspec error, not just 1 on "differences present." `grep` exits 2 on file-read errors (permission denied, no such file) — distinct from exit 1 ("no match"). `test` and `[` exit 2 on syntax errors. The proposal returns `EventType.SUCCESS` and `needs_llm=False` for *any* non-zero exit on these commands, conflating "informational non-match" with "the tool itself broke." A subtask that silently passes through a grep permission error as success will produce a downstream wrong-answer with no escalation. This is a correctness regression masquerading as a token-cost win, and the bench cannot catch it because the same author is writing the bench.

WEAKNESS-4 [MAJOR]: `shlex.split(last_command)[0]` is the wrong key for half the whitelist. — The set contains multi-token entries (`"git diff --quiet"`, `"git diff --exit-code"`, `"cmp -s"`, `"diff -q"`) but the lookup is described as "matched against `shlex.split(last_command)[0]` plus a tiny suffix check for `git diff --quiet`." The "tiny suffix check" handwave hides that `cmp -s file1 file2` and `diff -q a b` and `grep -q pattern file` all need flag-aware parsing, and `git diff --quiet -- path/to/file` only matches if the suffix check is permissive enough to also misfire on `git diff --quietly-not-really`. The "~25 lines" budget is doing a lot of work papering over a parsing problem the candidate hasn't actually solved. Real-world invocations include `command git diff --quiet`, `\git diff --quiet`, env-prefixed variants, and pipelines — none of which `shlex.split(...)[0]` handles.

WEAKNESS-5 [MAJOR]: The "if it wins, the seam writes itself later from two known-good strategies" argument is exactly the same prematurity it accuses the incumbent of. — Round 4 will face the question "we now have CmdAware and Legacy, do we abstract?" with the same epistemic position the incumbent had pre-experiment: one mature classifier and one new one. The candidate has not demonstrated that the abstraction is easier to see post-hoc; it has only deferred the decision by one round. Worse, by inlining the command-aware logic directly into `observation.py` (step 1), the eventual seam now has to *retroactively extract* logic from a tested, merged module, which is strictly harder than the incumbent's greenfield seam. The candidate confuses "defer the seam" with "make the seam cheaper."

WEAKNESS-6 [MINOR]: "interactive_runner.py:352 and toolcall_runner.py:246 already have the command string inline" is asserted without verification, and even if true, ignores `script_runner.py` and `planned_runner.py` which per CLAUDE.md are the "cheapest mode that works" defaults. — If the command-aware classifier only fires in interactive/toolcall paths, then `script` mode (which CLAUDE.md says the planner prefers for deterministic single-step work, exactly where `git diff --quiet` lives) gets none of the benefit. The 30% number is then measured on the modes least likely to be used for the targeted command shapes, which would either nullify the win or force the change into more call sites than the "2 lines" budget admits.

WEAKNESS-7 [MINOR]: Deferring the decision log is described as a virtue but the candidate offers no plan for *when* it gets built. — "Opt-in real-traffic logging is a separable feature whose design should be driven by a concrete classifier deficit the bench couldn't surface" is plausible-sounding but circular: the bench is hand-curated by the author of the classifier, so it cannot, by construction, surface a deficit the author didn't anticipate. The mechanism by which the "concrete classifier deficit" would ever be discovered is left unspecified, which means in practice the log gets built never, and the classifier evolves entirely against scenarios its own author writes.

VERDICT: A tightly-argued tactical PR proposal whose falsifiability theatre conceals one correctness regression (FATAL: silent swallowing of exit-2 tool errors), train/test contamination in the eval, and a hand-picked 30% threshold — the seam-vs-experiment framing is rhetorically sharp but doesn't survive contact with grep's exit codes.
